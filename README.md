# WasmOS

**A WebAssembly runtime management platform built by Team CTRL-5.**

WasmOS lets you upload, execute, inspect, and monitor WebAssembly modules through a REST API and a real-time web dashboard. The backend is written in Rust on top of Actix-Web; the frontend is a Next.js 14 app that talks to it over a simple proxy in development and gets baked into a static bundle for production.

We built this as our capstone project to explore what it would look like to treat WASM binaries the way an operating system treats processes — with a scheduler, resource isolation, capability-based access control, and full observability.

## What's in the box

| Folder | What it does |
|---|---|
| `wasmos/` | Rust backend — API server, WASM interpreter, scheduler, WebSocket events |
| `frontend/` | Next.js dashboard — task management, traces, snapshots, security analysis, terminal |
| `web/` | Static export of the dashboard (generated by `npm run build`, don't edit by hand) |
| `k8s/` | Kubernetes manifests if you want to go that route |
| `grafana/`, `prometheus.yml` | Pre-built Grafana dashboard and Prometheus scrape config |
| `docs/` | Academic deliverables — sprint reflections, design docs, test plans |

## How it works

During **development** you run two processes: the Rust backend on port 8080 and the Next.js dev server on port 3001. The dev server proxies all `/v1/*` and `/v2/*` requests to the backend so you don't have to think about CORS.

For **production** (or production-style local testing), you build the frontend into a static export that lands in `web/`. The Rust server picks that up and serves the UI and the API from a single origin — no separate frontend hosting needed.

## The dashboard

The web UI has grown quite a bit over the sprints. Here's what's in there:

- **Tasks** — upload `.wasm`/`.wat` files, run them, see stdout and return values, manage priorities. Also includes environment simulation tools (ABI mocking, a virtual filesystem explorer, runtime environment variables, and a scenario orchestrator for chaos testing).
- **Terminal** — a full xterm.js terminal that connects to the backend over WebSocket.
- **Traces** — distributed tracing with waterfall charts, environment-aware heatmaps, forensic crash snapshots, and a "clone to test" feature for building regression suites.
- **Snapshots** — capture and restore WASM execution state (memory, stack, globals). Supports time-travel diffing, environment sidecar metadata, memory-to-vFS mapping, and forking a snapshot with modified inputs.
- **Security** — static binary analysis, control-flow graph visualization, entropy heatmaps, import/export inspection, and YARA-style pattern matching.
- **Metrics** — live Prometheus metrics, system stats, per-task performance breakdowns.
- **Audit Log** — every API action is recorded with user, role, IP, and timestamp. The UI supports playback and filtering.
- **RBAC / Tokens** — capability-based access tokens with granular permissions and expiry.
- **Batch Execution** — run multiple WASM files in one request and get a combined report.
- **Monitor** — real-time system health and scheduler status.
